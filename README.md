# Fault-Tolerant Distributed Computation — OMNeT++

A discrete-event simulation of a **fault-tolerant distributed computing network** built with OMNeT++ 6.1. The system models a client-server architecture where clients distribute computational tasks across servers, detect and tolerate malicious nodes using **majority voting**, score server reliability, and propagate scores via a **gossip protocol** to optimize future task assignments.

---

## Overview

The network consists of **Client nodes** and **Server nodes** communicating over a configurable topology. Each client divides a task (finding the maximum element in an array) into `n` subtasks and dispatches them to `n/2 + 1` randomly selected servers. Servers may behave **honestly** (return correct results) or **maliciously** (return incorrect results). Clients use majority voting to determine the correct answer, score each server, and share scores via gossip. In the second round, task assignment is optimized by selecting only the top-ranked servers.

### Key Features
- **Majority voting** for fault-tolerant result aggregation across `n/2 + 1` servers
- **Malicious node simulation** — up to `n/4` servers can act maliciously per task
- **Server scoring system** — clients score servers based on response correctness
- **Gossip protocol** — scores are broadcast across all clients after each task round
- **Two-round execution** — round 1 uses random server selection; round 2 uses top-scored servers
- **Configurable topology** via `topology.txt` — no recompilation needed
- **Detailed logging** — client, server, and gossip activity written to separate log files

## Configuration

### `topology.txt`
Edit this file to configure the network. **Do not modify any source code.**

```
parameters
{
clients = 2
servers = 9
}

$client -> server delay
{
[100,100,200,100,100,200,100,100,200,]
[100,100,300,100,100,200,100,100,200,]
}

$server -> client delay
{
[100,100,200,100,100,200,100,100,200,]
[100,100,300,100,100,200,100,100,200,]
}

$client -> client delay
{
[0,100]
[200,0]
}
```

- `clients` / `servers` — number of nodes in the network
- Delay matrices define per-link delays in **milliseconds**
- Matrix dimensions must match the declared client/server counts

### `config.h`
Controls the number of task rounds per simulation run:
```cpp
const int Total_Task_Count = 2;
```

## How to Run

1. **Install OMNeT++ 6.1** and ensure it is on your system PATH.

2. **Install Python 3.8+** if not already available:
   ```bash
   python3 --version
   ```

3. **Edit topology** (optional):
   ```bash
   nano topology.txt
   ```

4. **Generate the NED file** using the network manager script:
   ```bash
   python3 networkManager.py
   ```
   When prompted, enter the number of tasks to schedule (integer).

5. **Open the project in OMNeT++** and click **Run**, or build and run from the terminal:
   ```bash
   make
   ./networksAssn2 -u Cmdenv omnetpp.ini
   ```

6. **Check output logs** after simulation:
   ```bash
   cat logs/client.log
   cat logs/server.log
   cat logs/gossip.log
   ```

> ⚠️ Always run `networkManager.py` **before** re-running the simulation in OMNeT++ to ensure the NED file is up to date with the topology.

## How It Works

### Round 1 — Random Server Selection
1. Each client generates a random integer array and divides it into `n` subtasks.
2. `n/2 + 1` servers are selected randomly.
3. Each subtask is sent to all selected servers. Up to `n/4` servers are flagged malicious and return the minimum instead of the maximum.
4. The client collects responses and applies **majority voting** to determine the correct answer per subtask.
5. Servers are scored: +1 for a correct response, 0 for incorrect.
6. After task completion, each client **gossips** its server scores to all other clients.
7. Clients aggregate received scores and compute a global server ranking.

### Round 2 — Score-Based Server Selection
1. Each client selects the **top `n/2 + 1` servers** by accumulated gossip score.
2. Tasks are executed again with the optimized server set, reducing the chance of malicious interference.

## Message Types

| Message | Description |
|---|---|
| `subtask` | Client → Server: carries subarray data and malicious flag |
| `subtaskResp` | Server → Client: carries computed result (max or min if malicious) |
| `gossip` | Client → Client: carries server scores after task completion |
